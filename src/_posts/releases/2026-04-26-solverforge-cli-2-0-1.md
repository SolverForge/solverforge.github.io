**solverforge-cli 2.0.1** is now available on
[crates.io](https://crates.io/crates/solverforge-cli/2.0.1) with API docs on
[docs.rs](https://docs.rs/solverforge-cli/2.0.1).

This is a scaffold-target refresh for the 2.0 CLI line. The public command
surface stays the same: `solverforge new` creates one neutral app shell, and
the `generate` commands grow the domain after the project exists.

## Install

```bash
cargo install solverforge-cli --version 2.0.1 --force
solverforge --version
```

The published binary now reports:

```text
solverforge solverforge-cli 2.0.1
CLI version: 2.0.1
Scaffold runtime target: SolverForge crate target 0.9.1
Scaffold UI target: solverforge-ui 0.6.3
Scaffold maps target: solverforge-maps 2.1.3
Runtime source: crates.io: solverforge 0.9.1
UI source: crates.io: solverforge-ui 0.6.3
Maps source: crates.io: solverforge-maps 2.1.3
```

## Scaffold Targets

Fresh projects generated by 2.0.1 start on:

- `solverforge 0.9.1`
- `solverforge-ui 0.6.3`
- `solverforge-maps 2.1.3`
- Rust `1.95`

The generated `Cargo.toml` now includes the current dependency line:

```toml
rust-version = "1.95"

[dependencies]
solverforge = { version = "0.9.1", features = ["serde", "console", "verbose-logging"] }
solverforge-ui = { version = "0.6.3" }
solverforge-maps = { version = "2.1.3" }
```

The matching app metadata records the same public target:

```toml
[app]
name = "my-scheduler"
starter = "neutral-shell"
cli_version = "2.0.1"

[runtime]
target = "solverforge 0.9.1"
runtime_source = "crates.io: solverforge 0.9.1"
ui_source = "crates.io: solverforge-ui 0.6.3"
```

## What Changes For Existing Docs

Before 2.0.1, generated apps needed an explicit post-scaffold bump when a
tutorial wanted the `solverforge 0.9.1` or `solverforge-ui 0.6.3` patch lines.
That caveat is gone for fresh projects generated by the current CLI.

Keep using `solverforge --version` as the source of truth for installed scaffold
targets. The CLI package version and generated-app dependency targets are
reported separately on purpose.
