**SolverForge 0.9.1** is now available on
[crates.io](https://crates.io/crates/solverforge/0.9.1), with API docs on
[docs.rs](https://docs.rs/solverforge/0.9.1), and is tagged in the
[core repository](https://github.com/SolverForge/solverforge/tree/v0.9.1).

This is a focused runtime release. The 0.9.0 line made the model contract
explicit with `planning_model!`, scalar/list terminology, and manifest-owned
runtime metadata. 0.9.1 keeps that public model shape and tightens what happens
inside scoring and local search.

## What changed

### Exact `usize` existence keys use indexed storage

`if_exists(...)` and `if_not_exists(...)` still read the same way in application
constraints: ask whether a matching record exists in another collection, then
penalize or reward the result.

Internally, exact `usize` join keys now use dense indexed bookkeeping for direct
and flattened existence constraints. Other key shapes, including
`Option<usize>`, newtype IDs, strings, and composite keys, keep hashed storage.
That preserves the public stream API while making common route-ID and
assignment-ID checks cheaper.

This matters for examples such as delivery routing, where a rule often flattens
all vehicle routes and checks whether each delivery ID appears in that flattened
stream.

### Local-search phase starts show the score

Console output now includes the current score when the local-search phase
starts. That score is calculated before local search begins, so the console can
show exactly what construction handed to local search:

```text
0.002s ▶ Local Search started │ 0hard/-50soft
```

The next progress line is now easier to read because the starting point is
visible without waiting for the first accepted move or progress tick.

### List ruin skips empty owners

The list-ruin selector now samples only entities whose list is non-empty. A
vehicle with no visits, or a machine with no queued jobs, should not spend a
ruin candidate on removing zero elements.

Empty owners can still receive elements during ordinary list-change,
sublist-change, or rebuild-style search. This change only removes wasted ruin
attempts from the candidate stream.

## Upgrade

For crates.io projects, update the SolverForge runtime target to `0.9.1`:

```toml
[dependencies]
solverforge = { version = "0.9.1", features = ["serde", "console"] }
```

For source or path-based development, use the `v0.9.1` tag when you want these
runtime changes.

For CLI-generated projects, keep `solverforge --version` as the source of truth
for the exact scaffold targets baked into the installed CLI.
`solverforge-cli 2.0.1` scaffolds `solverforge 0.9.1`, so newly generated apps
already start on this runtime patch line.

The public API shape from 0.9.0 remains the important modeling boundary:
`planning_model!` owns the model manifest, non-list planning variables are
scalar variables, and list variables remain the route or sequence surface for
ordered planning.
