**solverforge-cli 2.0.2** is now available on
[crates.io](https://crates.io/crates/solverforge-cli/2.0.2) with API docs on
[docs.rs](https://docs.rs/solverforge-cli/2.0.2).

This is a scaffold-target refresh for the current public SolverForge runtime.
Newly generated apps start on:

- `solverforge 0.10.0`
- `solverforge-ui 0.6.4`
- `solverforge-maps 2.1.3`
- Rust `1.95`

## Verify The Target

```bash
cargo install solverforge-cli --version 2.0.2 --force
solverforge --version
```

Expected output:

```text
solverforge solverforge-cli 2.0.2
CLI version: 2.0.2
Scaffold runtime target: SolverForge crate target 0.10.0
Scaffold UI target: solverforge-ui 0.6.4
Scaffold maps target: solverforge-maps 2.1.3
Runtime source: crates.io: solverforge 0.10.0
UI source: crates.io: solverforge-ui 0.6.4
Maps source: crates.io: solverforge-maps 2.1.3
```

## Generated Dependency Shape

Fresh projects generated by 2.0.2 include:

```toml
[dependencies]
solverforge = { version = "0.10.0", features = ["serde", "console", "verbose-logging"] }
solverforge-ui = { version = "0.6.4" }
solverforge-maps = { version = "2.1.3" }
```

```toml
[runtime]
target = "solverforge 0.10.0"
runtime_source = "crates.io: solverforge 0.10.0"
ui_source = "crates.io: solverforge-ui 0.6.4"
maps_source = "crates.io: solverforge-maps 2.1.3"
```

The current published `solverforge-ui` crate is `0.6.5` for create-job
identifier normalization. Use that patch line directly in app-owned manifests
when you need the 0.6.5 fix before the CLI scaffold target moves again.

The `solverforge-maps` source repository is tagged `v2.1.4` for matrix
route-distance access. `solverforge-cli 2.0.2` still scaffolds
`solverforge-maps 2.1.3`; use the maps source tag directly in app-owned
manifests when you need the 2.1.4 matrix-distance API before a newer CLI
scaffold target is released.
